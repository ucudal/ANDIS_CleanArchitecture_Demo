117DEFAULT_SCHEMA = """\
118<< Structured Request Schema >>
119When responding use a markdown code snippet with a JSON object formatted in the following schema:
120
121```json
122{{{{
123 "query": string \\ text string to compare to document contents
124 "filter": string \\ logical condition statement for filtering documents
125}}}}
126```
127
128The query string should contain only text that is expected to match the contents of documents. Any conditions in the filter should not be mentioned in the query as well.
129
130A logical condition statement is composed of one or more comparison and logical operation statements.
131
132A comparison statement takes the form: `comp(attr, val)`:
133- `comp` ({allowed_comparators}): comparator
134- `attr` (string): name of attribute to apply the comparison to
135- `val` (string): is the comparison value
136
137A logical operation statement takes the form `op(statement1, statement2, ...)`:
138- `op` ({allowed_operators}): logical operator
139- `statement1`, `statement2`, ... (comparison statements or logical operation statements): one or more statements to apply the operation to
140
141Make sure that you only use the comparators and logical operators listed above and no others.
142Make sure that filters only refer to attributes that exist in the data source.
143Make sure that filters only use the attributed names with its function names if there are functions applied on them.
144Make sure that filters only use format `YYYY-MM-DD` when handling date data typed values.
145Make sure that filters take into account the descriptions of attributes and only make comparisons that are feasible given the type of data being stored.
146Make sure that filters are only used as needed. If there are no filters that should be applied return "NO_FILTER" for the filter value.\
147"""
148DEFAULT_SCHEMA_PROMPT = PromptTemplate.from_template(DEFAULT_SCHEMA)
